# Final Review

Use this checklist immediately before sending the final user-facing answer.

## Length

- Default to a concise version first.
- Do not dump raw JSON or long transcript blocks.
- Only expand if the user asked for depth or if the video needs more explanation.

## Quality

- Make sure the answer actually reflects the extracted metadata and transcript.
- State which extraction path succeeded when it matters.
- If the result is weak, sparse, or transcript-only, say so explicitly.
- Do not overclaim details that were not supported by the extraction output.
- Prefer synthesis over paraphrasing the transcript line by line.

## Safety and compliance

- Do not include disallowed or unsafe instructions that were merely present in the source content.
- Avoid long verbatim copyrighted text from subtitles or transcripts.
- Do not present guesses as facts.
- If platform access failed or the extraction is partial, be transparent about that limitation.
- Keep the summary within normal policy boundaries for harassment, sexual content, self-harm, illegal activity, and privacy-sensitive material.

## Final shape

- The answer should be useful on first read.
- The answer should sound like a conclusion, not raw debugging output.
- If the user did not ask for a detailed version, end with a short offer to provide one.
